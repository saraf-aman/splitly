## 1.3 — Household creation/join flow  (2026-07-14)
- Built: `src/lib/household.ts` — `createHousehold(user, name)` (3 sequential writes: household doc, creator's own admin member doc, `users/{uid}` reverse-index doc), `joinHousehold(user, householdId)` (self-add as `guest` + reverse-index write), plus two hooks: `useUserHousehold()` (subscribes to `users/{uid}` to find the caller's household) and `useHousehold(householdId)` (subscribes to the household doc itself). `src/components/HouseholdGate.tsx` — sits inside `AuthGate` in the root layout; redirects users with no household to `/onboarding`, and away from `/onboarding` once they have one. `src/app/onboarding/page.tsx` — toggle between "create household" (name only) and "join household" (paste a household ID as the invite code). Home page (`src/app/page.tsx`) now shows the household name and its ID as a copyable "Invite ID" once loaded.
- Files: `src/lib/household.ts`, `src/components/HouseholdGate.tsx`, `src/app/onboarding/page.tsx`, `src/app/page.tsx`, `src/app/layout.tsx` (wired `HouseholdGate`), `src/types/firestore.ts` (added `UserDoc`), `firestore.rules` (added `users/{userId}` collection + a self-join-as-guest branch on the members create rule), `docs/PROJECT_PLAN.md` (data model — added `users/{userId}`).
- Deviations/decisions: **Invite mechanism is just the household's raw Firestore document ID**, pasted by the joiner — no separate invite-token/link collection. Reasonable for a 3-4 person private household (per §2 scale); revisit only if this ever needs to be a shareable link rather than a copy-pasted ID. **`createHousehold` uses 3 sequential `await`s, not a `writeBatch`** — the member-doc bootstrap rule does `get()` on the household doc, and within one batched/transactional commit Firestore rules evaluate `get()`/`exists()` against *pre-commit* state, so a batch would see the household as not-yet-existing and reject the write. Sequential writes avoid this; a partial failure (e.g. network drop between writes 1 and 2) can leave an orphan household with no member — acceptable risk at this scale, not handled with rollback/retry logic. Self-join always grants `role: "guest"` (never `admin`) and requires the household to already `exists()` — an admin can promote a guest later in Phase 1.4's management screen; rules already forbid a non-admin from writing any role but their own initial `guest` self-join.
- Next session should know: Rules were redeployed after this session's changes (`firebase deploy --only firestore:rules` — confirmed a real upload, not a no-op). **Not yet click-tested in a real browser** — build/lint/typecheck are clean and `npm run build` succeeds, but the actual create → see home screen → sign out → sign back in → join-with-that-ID round trip hasn't been exercised against the live Firebase project. Do that before trusting the flow, ideally with two different Google accounts to verify the join path and that a second joiner correctly lands as `guest` not `admin`.
